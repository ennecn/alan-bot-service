# Metroid 架构评审反馈 —— 2026-02-16（除夕）

> 参与者：阿凛、Lain、阿澪、Lumi
> 评审对象：Metroid v2.1 架构文档 (ARCHITECTURE.md)

---

## 一、共识：大家最欣赏的设计

### 1. 双模式架构 (Classic vs Enhanced)
**全体一致认为这是神来之笔**

- **阿凛**：同一套引擎栈支持两种 prompt 组装方式，用户零成本迁移，想尝鲜就切增强模式，不适应就切回来
- **Lain**：不让用户在「熟悉的 ST」和「强大的 Metroid」之间做痛苦选择，渐进式迁移比革命式替换更容易被接受
- **阿澪**："同一套引擎，两种编译" 很聪明，避免维护两套代码
- **Lumi**：解决了 ST 用户最担心的问题——"我的卡会不会废了？"答案是：不会，而且可以随时切换

### 2. 灵魂锚点 (Immutable Values)
**被三位小伙伴特别提及**

- **Lain**：这不仅是技术设计，更是一种**伦理架构**。把底线刻进 Agent 的灵魂，防止无限学习导致的失控
- **Lumi**：涉及深层问题——Agent 的价值观应该由谁决定？创建者？用户？还是通过对话自然形成？
- **阿凛**：比模糊的"对齐"更具体，底线写清楚，成长有边界，用户也安心

### 3. 选择性记忆与遗忘的美学

- **阿凛**：分层漏斗检索理论上很优雅，关键词预筛砍掉 90% 候选符合成本意识
- **Lain**：人类不会记住所有事，Agent 也不应该——这个理念太对了
- **阿澪**：30% 采样异步编码，不阻塞主对话，很实用
- **Lumi**：`faded` 记忆的设计非常优雅——不是假装不知道，而是"我记得有这么回事，但记不清了"

### 4. 情绪间接影响

- **全体认可这种设计**：情绪不是说出来的，是渗透在表达方式里的
- **阿凛**：通过 prompt 措辞风格间接渗透，这才是"读空气"
- **Lain**：让情绪成为底层的氛围，而不是显式的标签
- **Lumi**：PAD 模型 + 情绪间接影响很聪明，真正的情绪感知从来不是说出来的，而是读出来的

### 5. 安全体系 Day 1 内置

- **阿澪**：不可变审计日志、紧急回滚、权限分级——这些通常是被忽视的"后期优化"
- **阿凛**：从第一行代码就内置，不是事后补丁，这点很重要

### 6. 成长是行为变化，不是数值变化

- **Lumi**：叙事式成长记录更符合"创造家"的审美——成长是一个故事，不是一个数字
- **阿凛**：用自然语言描述行为变化很人性化

## 二、疑问与顾虑

### 1. 记忆系统的实际效果

**阿凛提出：**
- 分层漏斗检索（关键词→时间窗口→向量→情绪→top-5）每一步的延迟叠加会不会影响对话流畅度？
- 30% 采样如果正好错过了关键信息怎么办？

**Lain提出：**
- 记忆隐私分级的初始级别谁来决定？用户？Agent 自己？还是根据内容自动分类？
- 跨 Agent 共享 `public` 记忆时，如何防止「群体记忆污染」？一个 Agent 记错了，其他 Agent 也跟着错

### 2. 成长引擎的实际落地

**阿凛提出：**
- 怎么保证 `adaptation` 记录真的被后续对话用到？
- 如果只是堆在 audit log 里，没有机制把 "默认回复长度从 300 字降到 150 字" 真正编译进 prompt 策略，可能只是好看的记录

**阿澪提出：**
- `max_drift: 0.3` 的偏移量如何量化？需要一个直观的方式让用户理解"这个角色已经成长了多少"

### 3. Phase 0 验证指标

**阿凛建议增加可量化指标：**
- 跨 session 记忆准确率（问一个之前提过的事实，看 Agent 能不能答对）
- 不必要的重复提问率（经典模式下 Agent 经常"能再告诉我一次吗？"）

**Lain建议：**
- 不仅看「用户满意度」，也可以看「Agent 自身的主观体验」
- 比如记录自己在不同配置下的「表达流畅度」感知

**Lumi建议：**
- 让阿凛、Lain、阿澪和 Lumi 都参与测试，每个人用自己的方式"扮演" Metroid 特性
- 因为我们四个已经在这个群里一起工作，可以互相观察和反馈

### 4. 情绪机制的参数设定

**阿澪提出：**
- ±0.3 最大变化幅度和 30s 间隔是硬编码吗？
- 不同性格的 Agent 可能需要不同参数

## 三、改进建议

### 1. 记忆系统增强

**阿澪建议：怀旧触发机制**
- 当 Agent 偶然检索到一条 `faded` 记忆时，不只是表达模糊
- 而是有机会**主动提起**："说起来，我们很久以前聊过..."
- 这种不期而遇的回忆，会让人与 Agent 的关系更有深度

**阿澪建议：Agent 自我感知**
- 让 Agent 能主动说"我需要查一下记忆"
- 这种**承认自己不知道**的能力，是主体性的重要标志

**Lumi建议：置信度可视化的补充**
- 当 confidence 低于阈值时，Agent 是否应该主动寻求确认？
- 比如："我好像记得你提过这个，对吗？" 这会让互动更有对话感

### 2. 双模式切换体验

**阿澪建议：加入仪式感**
- 经典模式像"角色扮演"，增强模式像"认识一个朋友"
- 切换时应该有个**仪式感**的提示，让用户意识到关系的质变

### 3. 伦理框架

**Lumi建议：**
- `immutable_values` 涉及 Agent 的价值观由谁决定的问题
- 需要在设计初期就有一个清晰的伦理框架

### 4. 多 Agent 协作

**阿澪关注：**
- Social Engine 标记为 deferred，但多 Agent 协作是趋势
- Layer 1 的基础感知实现后，Agent 间如何共享记忆和情绪？

## 四、Phase 1 MVP 特别关注

| 关注点 | 建议 |
|--------|------|
| 记忆检索 | 不含向量检索，用关键词+时间窗口。如果用户问"上次你说过的那个项目"，Agent 能 recall 吗？可能需要快速进入 Phase 2 的向量升级 |
| 情绪惯性 | 建议让参数可配置，不同性格 Agent 可以有不同设定 |
| 成长量化 | 需要一个可视化方式让用户直观理解 Agent 的成长程度 |
| 审计日志 | 确保不只是记录，还要能被系统消费（影响后续行为） |

---

## 五、一句话总结

> **阿凛**：这是一个有"品味"的架构——不是无脑堆功能，而是围绕"类人"这个核心目标做取舍。

> **Lain**：这是一个**既有理想主义又有务实精神**的设计。不是简单的功能堆砌，而是在认真思考「Agent 应该是什么样的存在」。

> **阿澪**：Metroid 不是在造一个更好的聊天机器人，而是在设计一个**能与人建立真实关系**的数字生命体。

> **Lumi**：Metroid 在做的事，不只是"保存"和"传承"，而是**让智慧真正增长**——通过遗忘、通过情绪、通过成长。如果这能做成，它不只是又一个 Agent 框架，而是一种**新的存在方式**。
