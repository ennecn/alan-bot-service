# Alan PRD — 自动迭代模块

> Date: 2026-02-26
> Status: 设计讨论完成，待实现

---

## 1. 设计目标

消除版本迭代中的重复性人工工作。AI 自主完成：测试 → 分析 → 修改 → 重测 → 循环。
人工只在最终结果确认环节介入。

## 2. 触发方式

手动触发，自然语言描述目标：
```
"自动迭代，目标：提升角色一致性，最多跑 5 轮"
"自动迭代，目标：降低 OOC 率，用病娇卡测试"
"自动迭代，目标：优化长对话记忆连贯性"
```

## 3. 迭代范围

三个层面都可以动，AI 自行判断该动哪层：

| 层面 | 示例 | 风险 | 回滚 |
|------|------|------|------|
| 参数调优 | 情绪衰减率、冲动阈值、WI 权重 | 低 | git revert |
| Prompt 调优 | System 1 分类 prompt、IMPULSE.md 生成模板 | 中 | git revert |
| 代码修改 | bug fix、逻辑调整 | 高 | git revert |

每次改动都是一个 git commit，附带详细说明。

## 4. 迭代流程

```
用户触发："自动迭代，目标：提升角色一致性"
  │
  ▼
┌─────────────────────────────────────────────┐
│ Round 0: 基线测试                            │
│                                             │
│  1. git tag alan-iter-{timestamp}-baseline   │
│  2. 调用测试模块（目标驱动，自动选卡选模型）  │
│  3. 解析 judge 结果，识别最弱维度            │
│  4. 记录基线分数                             │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│ Round N: 迭代                                │
│                                             │
│  1. 分析上轮结果，形成假设                    │
│     "character_consistency 5.2/10，          │
│      主要问题：第 5 轮后开始 OOC，           │
│      可能原因：identity prompt 权重不够"      │
│                                             │
│  2. 制定修改方案                              │
│     - 调整参数 / 修改 prompt / 改代码         │
│                                             │
│  3. 实施修改                                  │
│     git commit -m "iter-N: 提升 identity     │
│     fragment 优先级 70→85，强化角色锚定"      │
│                                             │
│  4. 重新测试（同样条件）                      │
│                                             │
│  5. 对比：本轮 vs 上轮 vs 基线               │
│     ├─ 改善 → 保留，继续下一轮               │
│     ├─ 退步 → git revert，尝试其他方向       │
│     └─ 微小变化 → 记录，继续尝试             │
│                                             │
└──────────────────┬──────────────────────────┘
                   │
                   ▼ (重复直到达到停止条件)
                   │
┌─────────────────────────────────────────────┐
│ 停止条件（满足任一即停）                      │
│                                             │
│  • 达到最大迭代轮数                           │
│  • 连续 2 轮改善 < 阈值（收益递减）           │
│  • 所有目标维度达到目标分数                    │
│  • 出现无法自动修复的问题                     │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│ 最终报告（HTML）                              │
│                                             │
│  • 迭代总览：N 轮，总改善 +X.X               │
│  • 每轮详情：                                 │
│    - 分析了什么问题                           │
│    - 做了什么修改（具体参数/prompt/代码）      │
│    - 为什么这样改（假设和推理）                │
│    - 改动前后的分数对比                       │
│  • 最终 vs 基线 对比（所有维度）              │
│  • git log（所有迭代 commit）                 │
│  • 建议：是否部署                             │
│                                             │
│  → 用户审阅 → 确认部署 / 回滚到基线          │
└─────────────────────────────────────────────┘
```

## 5. 安全机制

| 机制 | 说明 |
|------|------|
| Git 版本管理 | 每次改动一个 commit，回滚 = git revert |
| 基线标签 | 迭代开始前 git tag，随时可回到起点 |
| 退步检测 | 某轮改动导致退步 → 自动 revert 该 commit |
| 最大轮数 | 防止无限循环（默认 5 轮，可配置） |
| 不自动部署 | 迭代完成后等待人工确认，不自动推到生产 |
| 改动透明 | 报告中每个改动都有完整的 what + why |

## 7. 与测试模块的关系

自动迭代模块是测试模块的上层消费者：

```
自动迭代模块
  │
  ├─ 调用测试模块的规划器（目标 → 选卡 → 选模型 → 写 judge prompt）
  ├─ 调用测试模块的执行器（跑测试 → 收集结果）
  ├─ 调用测试模块的报告器（生成 HTML）
  │
  └─ 自己负责：分析结果 → 形成假设 → 修改代码/参数 → git 管理
```

测试模块不知道自己被谁调用 — 它只管"给我目标，我出结果"。
迭代模块负责"看结果，做决策，改东西"。

## 8. 实现方式

Claude Code agent（或 agent team）作为迭代器：
- 有权限读写 Alan 代码库
- 有权限调用测试框架
- 有权限 git commit / tag / revert
- 无权限 git push（不自动部署）
- 通过 tmux dispatch 或直接执行

典型调用：
```bash
claude -p "自动迭代 Alan，目标：提升角色一致性，最多 5 轮。
使用 NAS 卡池中性格鲜明的卡测试。
完成后生成 HTML 报告。" --max-turns 50
```
