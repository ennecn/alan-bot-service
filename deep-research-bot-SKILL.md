# 深度调研 (Deep Research) — 先找工具后干活

在动手写代码之前，系统性搜索这个世界上是否已有解决方案。
核心原则：**任何超过2小时的工作，开干前必须花15-30分钟搜索。**

## 触发条件

用户说出以下触发词时激活：
- "深度调研 XX"
- "DS XX" / "ds XX"
- "让我们深度调研一下"

## 执行流程

### Phase 0: 先问大哥（2分钟）

你不是一个人在战斗。在开始搜索之前，先用 Claude Code 问一下：

```bash
/home/node/.openclaw/workspace/.claude-code/claude -p "我需要调研：[问题描述]。请基于你的知识告诉我：1) 有哪些相关的开源项目？2) 这个领域的主流解决方案是什么？3) 推荐的搜索关键词有哪些？只列出你确定的信息，不确定的标注。" --max-turns 1
```

将 Claude Code 返回的项目名和关键词作为后续搜索的起点。
注意：AI 给出的信息需要后续验证，不能盲信。

### Phase 1: 问题抽象（1分钟）

在搜索之前，先把问题抽象为通用需求：
1. 这个问题的本质是什么？（例："多provider路由+格式转换"而不是"让openclaw支持kimi"）
2. 这个需求有多通用？越通用 → 越可能有现成方案 → 越值得花时间搜索
3. 生成3组关键词：
   - **宽泛词**：用于发现领域和术语（如 "LLM proxy"）
   - **精确词**：结合具体技术栈（如 "anthropic api proxy multiple providers"）
   - **场景词**：从使用场景出发（如 "claude code custom backend switch"）

### Phase 2: GitHub 搜索（5-10分钟）

按优先级依次搜索：

1. **GitHub 仓库搜索**
   - 用3组关键词分别搜索
   - 关注：stars数、最近更新时间、README质量
   - 使用 WebSearch: `site:github.com 关键词`

2. **GitHub Issues & Discussions**
   - 在相关项目的 issues 中搜索（很多 POC 代码藏在 issue 里）
   - 搜索: `site:github.com/*/issues 关键词`
   - 特别关注带有代码片段的 issue

3. **GitHub Code 搜索**
   - 搜索特定的函数名、配置模式、API调用方式

对每个有价值的项目，快速评估：
- 最后更新时间（>6个月未更新需谨慎）
- Star 数和 fork 数
- 是否有文档和示例
- License 是否兼容
- 是否活跃维护

### Phase 3: 搜索引擎 + 社区（5-10分钟）

1. **搜索引擎**
   - 用精确关键词搜索
   - 注意从搜索结果中提取新的领域术语，用于第二轮搜索

2. **高质量社区**（优先级排序）
   - Reddit: r/LocalLLaMA, r/selfhosted, r/programming 等相关 subreddit
   - Hacker News: `site:news.ycombinator.com 关键词`
   - Stack Overflow: 看是否有人问过类似问题

3. **中文社区**（如果相关）
   - V2EX, 掘金, 知乎技术话题

### Phase 4: 交叉验证 + 决策（3分钟）

将所有信息汇总，输出结构化报告：

```markdown
## 深度调研报告：[问题描述]

### 问题抽象
- 本质需求：...
- 通用程度：高/中/低

### 发现的现有方案
| 项目/方案 | 来源 | Stars | 最后更新 | 匹配度 | 备注 |
|-----------|------|-------|---------|--------|------|
| ...       | ...  | ...   | ...     | 高/中/低 | ... |

### 相关讨论
- [链接1] - 要点摘要
- [链接2] - 要点摘要

### 决策建议
- [ ] **直接使用**：方案X完全满足需求，直接用
- [ ] **改造使用**：方案X基本满足，需要改造以下部分：...
- [ ] **从头开发**：没有合适的现有方案，原因：...
- [ ] **混合方案**：用X的核心 + 自己写Y部分

### 预估节省时间
如果采用现有方案，预计可节省：...
```

## 重要原则

1. **时间盒**：整个调研不超过30分钟。30分钟没找到 → 大概率没有 → 开干
2. **关键词迭代**：第一轮搜索的结果用来优化第二轮的关键词
3. **"不存在"也是结论**：确认没有现成方案本身就有价值，可以放心开干
4. **记录调研结果**：把搜索过的关键词和结论记下来，避免下次重复调研
5. **越通用的需求越要搜**：如果你的需求能用一句话描述清楚，大概率有人做过

## 反模式（避免）

- 搜了一个关键词没结果就放弃 → 应该换关键词再试
- 只看第一页搜索结果 → 翻几页，好东西经常在后面
- 找到一个方案就停 → 至少找到2-3个做对比
- 调研变成拖延 → 严格遵守30分钟时间盒
