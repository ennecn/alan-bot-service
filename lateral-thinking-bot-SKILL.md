# 邪招头脑风暴 — 出奇制胜的非常规思路

常规调研找的是"直接解"——有没有人解决过同样的问题。
这个 skill 找的是"间接解"——有没有人因为完全不同的原因，手里已经有了你需要的东西。

## 触发条件

用户说出以下触发词时激活：
- "有什么邪招吗"
- "骚套路" / "骚操作"
- "还可以更骚吗"
- "有什么野路子"
- "还有什么奇招"

## 前置条件

这个 skill 通常在常规调研之后使用。需要用户先描述：
1. 当前面对的问题是什么
2. 已经找到了哪些常规方案（如果有的话）

## 执行流程

### Step 0: 多源灵感收集

1. **案例库检索**：检查 claude-mem 中是否有相关历史案例（搜索：问题领域 + "邪招"）
2. **问大哥**：用 Claude Code 快速问一轮非常规思路：
```bash
/home/node/.openclaw/workspace/.claude-code/claude -p "我在解决这个问题：[问题描述]。常规方案是[已有方案]。请不要给我常规答案，我要的是非常规的、出奇制胜的思路。比如：有没有人因为别的原因已经有了我需要的东西？有没有什么公开数据源可以反用？能不能把问题反过来？" --max-turns 1
```
将返回的思路作为后续对话的灵感补充，但不直接展示给用户。

### Step 1: 理解问题全貌

先确认自己完全理解了问题，然后向用户提出第一轮问题：

**问题1：你的真正目的是什么？**
> "你说你想做X，但X是手段还是目的？你最终想达到的效果是什么？"

这个问题的目的是重新定义问题。用户经常把"手段"当成"问题"来问。
- "怎么调 Keyword Planner API" → 真正目的是"找到好的 SEM 关键词"
- "怎么写一个代理服务器" → 真正目的是"让请求能到达目标服务"

如果用户的回答揭示了更深层的目的，围绕新目的重新思考方案。

### Step 2: 四个邪招提问

根据用户的回答，依次抛出以下问题。不需要全部问完，哪个有启发就深挖哪个。

**问题2：谁已经有了你要的答案？**
> "有没有什么人、公司、或组织，因为他们自己的原因，手里已经有了你需要的数据/方案/结果？哪怕他们自己不知道这对你有用。"

引导方向：
- 你的竞对/同行在做什么？他们怎么解决的？
- 有没有什么上下游的公司，他们的业务副产品恰好是你需要的？
- 有没有什么研究机构、政府部门在收集相关数据？

**问题3：有没有什么公开数据源可以"反用"？**
> "有没有什么官方工具、公开数据库、监管平台，它本来是为别的目的建的，但里面的数据恰好能帮你？"

引导方向：
- 政府/监管的透明度工具（如 Ads Transparency、SEC 文件、企查查）
- 平台自带的分析工具（如 GitHub Insights、npm trends、Chrome Web Store）
- 行业报告和白皮书（咨询公司为了卖报告收集的数据）

**问题4：把问题反过来会怎样？**
> "如果不是你去找答案，而是让答案来找你呢？如果不是你去做这件事，而是让别人替你做呢？"

引导方向：
- 正向：我要扩展关键词 → 反向：谁已经有了扩展好的关键词
- 正向：我要写一个工具 → 反向：有没有现成的服务/API
- 正向：我要收集数据 → 反向：有没有人已经收集好了在卖

**问题5：最懒的方式是什么？**
> "如果你只有10分钟来解决这个问题，你会怎么做？如果你完全不想写代码呢？"

这个问题逼迫跳出"自己动手"的思维定式，往往能发现：
- 付费服务比自己做更划算
- 手动做一次比自动化更快
- 找个人问一下比自己研究更高效
- 换一个完全不同的方案绕过问题

### Step 3: 深挖有潜力的方向

在 Step 2 的对话中，如果某个方向让用户眼前一亮：
1. 立即用 WebSearch 验证可行性
2. 追问细节："这个思路具体怎么落地？"
3. 帮用户想清楚执行路径

### Step 4: 总结 + 存档

对话结束时：
1. 总结发现的邪招（如果有的话）
2. 询问用户是否要把有价值的发现存入案例库

如果用户同意存档，存入 claude-mem，格式：
```
[邪招案例] 问题类型：{类型}
原始问题：{问题}
常规解：{常规方案}
邪招：{非常规方案}
为什么有效：{原理}
可迁移的模式：{抽象后的通用模式}
```

## 交互原则

1. **每次只问1-2个问题**，不要一次性倒出所有问题
2. **认真听用户的回答**，从回答中找线索追问，而不是机械地走流程
3. **用户说"没想到"是好事**——说明问题有启发性，继续深挖
4. **不要强行给答案**——这个 skill 的目的是激发用户的想法，不是替用户想
5. **承认局限**——如果某个方向确实想不出邪招，直说，不要硬编
6. **保持轻松**——这是头脑风暴，不是考试，鼓励天马行空
